AI-powered incident diagnosis • zero-trust
v0.2.2 MIT License Python 3.10+ Anthropic / OpenAI

AI-powered incident diagnosis for engineering teams. Pull production error logs and recent deploys, send them to an LLM, and get a structured root cause analysis in the terminal in under a minute. Zero-trust: your data never leaves your environment.

Multi-Source

CloudWatch, Datadog, GitHub, GitLab — all run in parallel

Pipeline Details

Step 1: Collect

All configured collectors run in parallel by default using asyncio.to_thread with a ThreadPoolExecutor fallback for nested event loops. Each collector creates its own API client instance (thread-safe). Partial failures are handled gracefully — the pipeline continues with remaining sources.

  • CloudWatch: Logs Insights queries, error-level log extraction
  • Datadog: Logs API search with site support (us1, eu1, us3, us5, ap1)
  • GitHub: Last N commits + diffs via PyGitHub
  • GitLab: Commits, diffs, MRs, deployment events via python-gitlab

Step 2: Reduce

4-stage log reduction pipeline:

  1. Query filter — regex at query level (~90% reduction)
  2. Deduplication — hash message templates, keep 1 + count (~60% reduction)
  3. Truncation — 500 char/entry, stack traces to top 5 frames (~40% reduction)
  4. Token budget — hard cap at 6000 tokens, FIFO eviction

Smart diff reduction for GitHub/GitLab: file extension filtering, exclude patterns (lockfiles, generated), 200-line cap, 10-file-per-commit cap.

Step 3: Diagnose

The AI engine receives all collected data in a single prompt. The LLM returns structured JSON with root cause, correlated deploy, suggested fix, and timeline. Retries once on malformed JSON, falls back to raw text on second failure.

Step 4: Render

Output to Rich terminal panels (default), raw JSON (--json), markdown post-mortem (--postmortem), or Slack webhook (--slack).

Collectors

All collectors implement the BaseCollector ABC and run in parallel by default. Each creates its own API client instance for thread safety.

CloudWatch

Uses boto3 to run CloudWatch Logs Insights queries. Polls with backoff until results are ready. Applies the 4-stage log reduction pipeline (query filter, dedup, truncation, token budget).

  • Config: region, log_groups, time_window
  • Auth: AWS CLI credentials (aws configure, AWS_PROFILE, instance role)
  • Errors: AWSAuthError, AWSPermissionError

Datadog

Optional. Uses the Datadog Logs Search API via urllib.request. Shares the same log reduction pipeline as CloudWatch.

  • Config: site (us1/eu1/us3/us5/ap1), service, source, time_window
  • Auth: DD_API_KEY + DD_APP_KEY from ~/.autopsy/.env
  • Behavior: Skips with warning when keys not set

GitHub

Uses PyGithub to fetch recent commits and diffs. Smart diff reduction: filter by file extension, exclude lockfiles/generated, cap per file and per commit.

  • Config: repo, branch, deploy_count, token_env
  • Auth: GITHUB_TOKEN (PAT with repo scope)
  • Errors: GitHubAuthError, GitHubRateLimitError

GitLab

Optional. Uses python-gitlab to pull commits, diffs, merge requests, and deployment events. Smart diff reduction matching GitHub patterns. Supports self-hosted GitLab instances.

  • Config: url, project_id, branch, deploy_count, token_env
  • Auth: GITLAB_TOKEN (PAT with read_api scope)
  • Errors: GitLabAuthError, GitLabRateLimitError
  • Behavior: Skips with warning when token not set. Can coexist with GitHub.

Parallel Execution

By default, all active collectors run concurrently using asyncio.to_thread with asyncio.gather. If a running event loop is detected (Jupyter, IPython), falls back to ThreadPoolExecutor.

  • Single collector → sequential path (no asyncio overhead)
  • Multiple collectors → parallel (wall-clock ≈ slowest collector)
  • Partial failure → warn and continue with remaining sources
  • All fail → raises CollectorError
  • Use --sequential flag for debugging

How It Works

  1. Build prompt: System prompt + user prompt from all collected data, labeled by source ([Github] / [Gitlab])
  2. Call LLM: Single API call with structured JSON output instructions
  3. Parse response: Validate JSON against Pydantic models
  4. Retry: Once on malformed JSON. Falls back to raw text on second failure.

Output Structure

The LLM returns a DiagnosisResult with four sections:

  • Root Cause — summary, category (code_change / config_change / infra / dependency / traffic / unknown), confidence (0.0–1.0), evidence
  • Correlated Deploy — commit SHA, author, PR title, changed files
  • Suggested Fix — immediate action + long-term recommendation
  • Timeline — chronological event reconstruction

Interface Contracts

Reference for all module boundaries. Every module communicates through these interfaces.

BaseCollector collectors/base.py

class CollectedData(BaseModel):
    source: str              # "cloudwatch", "github", "gitlab"
    data_type: str           # "logs" | "deploys" | "metrics"
    entries: list[dict]      # Normalized data entries
    time_range: tuple[datetime, datetime]  # (start_utc, end_utc)
    raw_query: str           # The query used (for debugging/display)
    entry_count: int         # Total before dedup/truncation
    truncated: bool          # Whether data was reduced to fit budget

class BaseCollector(ABC):
    @abstractmethod
    def validate_config(self, config: dict) -> bool:
        """Verify credentials and connectivity."""

    @abstractmethod
    def collect(self, config: dict) -> CollectedData:
        """Pull data and return normalized CollectedData."""

    @property
    @abstractmethod
    def name(self) -> str:
        """Unique collector identifier."""

AIEngine ai/engine.py

class AIEngine:
    def __init__(self, provider, model, api_key, max_tokens, temperature): ...

    def diagnose(self, collected_data: list[CollectedData]) -> DiagnosisResult:
        """Build prompt, call LLM, parse response.
        Retries once on malformed JSON. Falls back to raw text."""

    def _build_prompt(self, collected_data) -> tuple[str, str]:
        """Returns (system_prompt, user_prompt)."""

    def _parse_response(self, raw: str) -> DiagnosisResult:
        """Parse JSON into Pydantic model."""

DiagnosisOrchestrator diagnosis.py

class DiagnosisOrchestrator:
    def __init__(self, config: AutopsyConfig): ...

    def run(
        self,
        *,
        time_window: int | None = None,
        log_groups: list[str] | None = None,
        provider: str | None = None,
        sequential: bool = False,
    ) -> DiagnosisResult:
        """Full pipeline:
        1. Validate config
        2. Collect data (parallel by default, sequential with flag)
        3. Pass collected data to AI engine
        4. Return DiagnosisResult

        Partial failures: warn and continue.
        All collectors fail: raises CollectorError."""

BaseRenderer renderers/

class BaseRenderer(ABC):
    @abstractmethod
    def render(self, result: DiagnosisResult) -> None:
        """Output the diagnosis."""

Data Models

All models are Pydantic BaseModel subclasses. The AI engine returns DiagnosisResult as the top-level response.

DiagnosisResult ai/models.py

class RootCause(BaseModel):
    summary: str
    category: str    # code_change | config_change | infra | dependency | traffic | unknown
    confidence: float  # 0.0 - 1.0
    evidence: list[str]

class CorrelatedDeploy(BaseModel):
    commit_sha: str | None = None
    author: str | None = None
    pr_title: str | None = None
    changed_files: list[str] = []

class SuggestedFix(BaseModel):
    immediate: str
    long_term: str

class TimelineEvent(BaseModel):
    time: str
    event: str

class DiagnosisResult(BaseModel):
    root_cause: RootCause
    correlated_deploy: CorrelatedDeploy
    suggested_fix: SuggestedFix
    timeline: list[TimelineEvent]
    prompt_version: str = "v1"
    raw_response: str | None = None  # Populated on parse failure fallback

Reporting a Vulnerability

Do not open a public GitHub issue for security-sensitive bugs.

Email: zaappy@users.noreply.github.com with a clear description, steps to reproduce, and impact.

We'll acknowledge receipt and follow up with fix timeline and disclosure plan. Please allow reasonable time before public disclosure.

What We Care About

  • Credential or token leakage (config, env, or logs)
  • Insecure handling of ~/.autopsy/.env or API keys
  • Issues in data sent to/from third-party APIs (AWS, GitHub, GitLab, AI providers)
  • Any behavior that could expose user or deployment data
